## From in-memory map to persistent store

In the [Go REST API tutorial](/blog/building-rest-apis-with-go), we built a complete CRUD API backed
by `store.Memory` — a `sync.RWMutex`-protected `map[string]Item`. It works perfectly for getting
started: zero dependencies, instant setup, and full testability without infrastructure.

But in-memory storage means every restart wipes your data. Multiple replicas can't share state.
There's no query language, no indexing, no constraints. As soon as you move beyond a single-instance
dev server, you need a real database.

This tutorial picks up right where the Go API post left off. We'll:

- Design a proper PostgreSQL schema for the `items` domain
- Write versioned migrations with `golang-migrate`
- Connect from Go using `pgx` — the high-performance Postgres driver
- Refactor `internal/store` into a `Repository` interface backed by `database/sql`
- Write integration tests that run against a real Postgres database
- Wire everything into the existing `chi` handlers — with zero handler changes
- Run the full stack — API + Postgres — with Docker Compose

By the end, your API will survive restarts, support concurrent access safely, and be ready for
production deployment. The handler code from the previous tutorial remains unchanged — that's the
power of coding to an interface.

## Designing the `items` schema

Before writing a line of Go, we need to decide what our data looks like in the database. The
`Item` struct from the in-memory store carries four fields:

```go
type Item struct {
    ID        string    `json:"id"`
    Name      string    `json:"name"`
    Price     float64   `json:"price"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}
```

A naive approach would mirror this struct directly into Postgres columns. But databases give us
tools that maps don't — let's use them.

### Choosing types

| Go field     | Naive SQL     | Better SQL                | Why                                     |
|-------------|---------------|---------------------------|-----------------------------------------|
| `ID string`  | `TEXT`        | `UUID DEFAULT gen_random_uuid()` | Unforgeable, sortable, collision-free |
| `Name`       | `TEXT`        | `VARCHAR(255) NOT NULL`   | Enforce length and non-null             |
| `Price`      | `DECIMAL`     | `NUMERIC(10,2) NOT NULL CHECK (price > 0)` | Exact arithmetic, no float drift |
| `CreatedAt`  | `TIMESTAMPTZ` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | Let the DB fill it in            |
| `UpdatedAt`  | `TIMESTAMPTZ` | `TIMESTAMPTZ NOT NULL DEFAULT now()` | Same auto-fill strategy         |

Three important decisions here:

**UUID instead of auto-increment.** Sequential IDs (`BIGSERIAL`) are simple, but they leak record
counts, are trivially guessable, and complicate sharding. UUIDs generated by `gen_random_uuid()`
(which requires the `pgcrypto` extension, available by default) give you globally unique identifiers
with no central coordinator.

**NUMERIC instead of `float8`.** Floating-point arithmetic is inexact. `0.1 + 0.2 != 0.3` in IEEE 754.
For anything involving money — prices, invoices, balances — use `NUMERIC(precision, scale)`. It
stores exact decimal values, at the cost of slightly more storage and CPU.

**`TIMESTAMPTZ` with defaults.** `TIMESTAMPTZ` normalises all timestamps to UTC while preserving
the original timezone. Setting `DEFAULT now()` means you don't need to supply timestamps in your
`INSERT` statements — Postgres fills them in.

### Adding `deleted_at` for soft deletes

Hard deletes (`DELETE FROM items WHERE id = $1`) are permanent. A buggy `DELETE` call, an
accidental curl, or a cascade from a parent row can destroy data you need for audit trails,
analytics, or customer support.

Soft deletes mark a row as "deleted" without removing it physically. The standard pattern adds
a nullable timestamp:

```sql
deleted_at TIMESTAMPTZ
```

The convention: `NULL` means active; any non-null value means deleted (and stores *when* it was
deleted). Your queries add `WHERE deleted_at IS NULL` to every `SELECT`, `UPDATE`, and `DELETE`
(which becomes an `UPDATE SET deleted_at = now()`).

We'll add this column to our schema. It costs one nullable timestamp column per row — negligible
for most workloads — and gives you an instant undo for accidental deletions.

### The complete schema

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE items (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    name       VARCHAR(255) NOT NULL,
    price      NUMERIC(10,2) NOT NULL CHECK (price > 0),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at TIMESTAMPTZ
);

CREATE INDEX idx_items_name ON items (name);
CREATE INDEX idx_items_deleted_at ON items (deleted_at) WHERE deleted_at IS NOT NULL;
```

The partial index on `deleted_at` (`WHERE deleted_at IS NOT NULL`) is a small optimisation:
Postgres skips rows that don't match the `WHERE` clause, keeping the index compact. It helps
when you occasionally query for deleted items (e.g., admin dashboards or data recovery).

The `name` index supports `WHERE name = $1` lookups, which is common in search endpoints. You
could also add a `pg_trgm` GIN index for fuzzy search, but that's beyond this tutorial's scope.

## Setting up `golang-migrate` and writing the first migration

Schemas evolve. Columns get added, indexes change, constraints are tightened. Running manual SQL
files in order is error-prone — you'll forget which script ran in staging but not production,
or whether that hotfix migration made it past the load balancer.

[golang-migrate](https://github.com/golang-migrate/migrate) solves this with **versioned migrations**:
each migration is a pair of numbered files — an `up` script and a `down` script. The tool tracks
which version has been applied in a special table (`schema_migrations`) and only runs newer ones.

### Installation

```bash
go install -tags 'postgres' github.com/golang-migrate/migrate/v4/cmd/migrate@latest
```

The `-tags 'postgres'` flag compiles in the Postgres driver. Without it, `migrate` can't connect
to Postgres databases.

Verify it's installed:

```bash
migrate -version
# v4.17.0
```

### Directory layout

Inside your project root (`go-api-demo/`):

```
migrations/
├── 000001_create_items_table.up.sql
└── 000001_create_items_table.down.sql
```

The naming convention is strict: `NNNNNN_description.{up,down}.sql`. The number is a sequential
version. `golang-migrate` sorts files lexicographically, so `000001` is version 1, `000002` is
version 2, and so on.

### The first migration

**`migrations/000001_create_items_table.up.sql`:**

```sql
CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE TABLE items (
    id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    name       VARCHAR(255) NOT NULL,
    price      NUMERIC(10,2) NOT NULL CHECK (price > 0),
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at TIMESTAMPTZ
);

CREATE INDEX idx_items_name ON items (name);
CREATE INDEX idx_items_deleted_at ON items (deleted_at) WHERE deleted_at IS NOT NULL;
```

**`migrations/000001_create_items_table.down.sql`:**

```sql
DROP INDEX IF EXISTS idx_items_deleted_at;
DROP INDEX IF EXISTS idx_items_name;
DROP TABLE IF EXISTS items;
```

The `down` migration reverses the `up` migration exactly — drop indexes first, then the table.
`IF EXISTS` prevents errors when the migration hasn't been fully applied.

### Running migrations

Set the database URL as an environment variable:

```bash
export DATABASE_URL="postgres://app:secret@localhost:5432/app?sslmode=disable"
```

Apply all pending migrations:

```bash
migrate -database "$DATABASE_URL" -path ./migrations up
# 1/u create_items_table (2.341ms)
```

Roll back the last migration:

```bash
migrate -database "$DATABASE_URL" -path ./migrations down 1
# 1/d create_items_table (1.892ms)
```

Check the current version:

```bash
migrate -database "$DATABASE_URL" -path ./migrations version
# 1
```

### Makefile helpers

Typing out the full `migrate` command every time gets old. Add these targets to your `Makefile`:

```makefile
.PHONY: migrate-up migrate-down migrate-create

DATABASE_URL ?= postgres://app:secret@localhost:5432/app?sslmode=disable

migrate-up:
	migrate -database "$(DATABASE_URL)" -path ./migrations up

migrate-down:
	migrate -database "$(DATABASE_URL)" -path ./migrations down 1

migrate-create:
	@read -p "Migration name: " name; \
	seq=$$(printf "%06d" $$(($$(ls migrations/*.up.sql 2>/dev/null | wc -l) + 1))); \
	migrate create -ext sql -dir ./migrations -seq "$$name"
```

Now:

```bash
make migrate-up          # apply all pending
make migrate-down        # roll back one
make migrate-create      # scaffold next migration pair
```

The `migrate-create` target auto-increments the sequence number so you don't overwrite existing
files. It uses `migrate create -seq` under the hood, which generates properly prefixed files.

## Connecting to PostgreSQL from Go with `pgx`

Go's standard library provides `database/sql` — a generic interface for SQL databases. But the
standard library doesn't include any database drivers; you pair it with a driver that implements
the `database/sql/driver` interfaces.

For Postgres, two drivers dominate:

| Driver | Package | Notes |
|--------|---------|-------|
| `lib/pq` | `github.com/lib/pq` | Original Postgres driver; in maintenance mode |
| `pgx` | `github.com/jackc/pgx/v5` | High-performance, actively maintained; preferred for new projects |

We'll use `pgx`. It supports the standard `database/sql` interface *and* a native
interface with lower overhead. For this tutorial, we'll stick with `database/sql` — it keeps
our repository implementation driver-agnostic.

### Installation

```bash
go get github.com/jackc/pgx/v5
go get github.com/jackc/pgx/v5/stdlib
```

The `pgx/v5` module is the core driver. `pgx/v5/stdlib` registers it as a `database/sql` driver
named `"pgx"`, making it available through `sql.Open("pgx", dsn)`.

### Opening a connection

```go
// internal/store/postgres.go
package store

import (
    "database/sql"
    "fmt"
    "time"

    _ "github.com/jackc/pgx/v5/stdlib"
)

func Open(dsn string) (*sql.DB, error) {
    db, err := sql.Open("pgx", dsn)
    if err != nil {
        return nil, fmt.Errorf("open db: %w", err)
    }

    db.SetMaxOpenConns(25)
    db.SetMaxIdleConns(5)
    db.SetConnMaxLifetime(5 * time.Minute)

    if err := db.Ping(); err != nil {
        return nil, fmt.Errorf("ping db: %w", err)
    }

    return db, nil
}
```

A few details worth understanding:

- **`_ "github.com/jackc/pgx/v5/stdlib"`** — the blank import side-effects the driver registration.
  Without it, `sql.Open("pgx", ...)` panics with "unknown driver."
- **`SetMaxOpenConns(25)`** — caps the connection pool at 25 concurrent connections. Postgres
  defaults to 100 total connections; 25 per replica leaves headroom for other services.
- **`SetMaxIdleConns(5)`** — keeps up to 5 idle connections alive in the pool, avoiding the
  cost of establishing new connections on every request.
- **`SetConnMaxLifetime`** — recycles connections after 5 minutes, preventing stale connections
  from lingering behind a load balancer or after a Postgres restart.
- **`db.Ping()`** — verifies the connection before returning, failing fast if Postgres is
  unreachable. Don't skip this: a silent `sql.Open` + first query failure is harder to debug.

### The DSN format

The DSN (`Data Source Name`) for `pgx` is a URL:

```
postgres://user:password@host:port/dbname?sslmode=disable
```

For production, set `sslmode=require` or `sslmode=verify-full`. For local development with Docker,
`sslmode=disable` is fine.

## Refactoring `internal/store` into a repository interface

The key insight: our HTTP handlers don't need to know whether items live in a `map[string]Item` or
a PostgreSQL table. They just need to `List`, `Get`, `Create`, `Update`, and `Delete`. We can
encode that contract as a Go interface.

### The Repository interface

```go
// internal/store/repository.go
package store

import (
    "context"
    "time"

    "github.com/google/uuid"
)

type Item struct {
    ID        uuid.UUID  `json:"id"`
    Name      string     `json:"name"`
    Price     float64    `json:"price"`
    CreatedAt time.Time  `json:"created_at"`
    UpdatedAt time.Time  `json:"updated_at"`
}

type Repository interface {
    List(ctx context.Context) ([]Item, error)
    Get(ctx context.Context, id uuid.UUID) (Item, error)
    Create(ctx context.Context, name string, price float64) (Item, error)
    Update(ctx context.Context, id uuid.UUID, name string, price float64) (Item, error)
    Delete(ctx context.Context, id uuid.UUID) error
}
```

Changes from the `Memory` struct:

1. **`context.Context`** is the first parameter to every method. Database calls can be cancelled
   or timed out through the context — essential for request-scoped operations and graceful shutdown.
2. **`uuid.UUID`** replaces `string` for IDs. Go's UUID library (`github.com/google/uuid`) gives
   us type-safe parsing and zero-value checks. The JSON serialisation stays the same (`"id"` field).
3. **Errors replace `(Item, bool)` returns.** The `Memory` store used a boolean to signal
   "not found." A repository interface that talks to a database should return meaningful errors
   (`ErrNotFound`, `ErrConflict`, etc.) so callers can differentiate between "doesn't exist" and
   "database is on fire."
4. **`Item.ID` is now a `uuid.UUID`**, not a `string`.

### Keeping the Memory store around

Don't delete `internal/store/memory.go`. It's useful for handler unit tests — you can inject
a `Memory` store into handlers without spinning up Postgres. We'll adapt it to implement `Repository`:

```go
// internal/store/memory.go (updated)
package store

import (
    "context"
    "fmt"
    "sync"
    "time"

    "github.com/google/uuid"
)

type Memory struct {
    mu    sync.RWMutex
    items map[uuid.UUID]Item
}

func NewMemory() *Memory {
    return &Memory{items: make(map[uuid.UUID]Item)}
}

func (m *Memory) List(_ context.Context) ([]Item, error) {
    m.mu.RLock()
    defer m.mu.RUnlock()

    items := make([]Item, 0, len(m.items))
    for _, item := range m.items {
        items = append(items, item)
    }
    return items, nil
}

func (m *Memory) Get(_ context.Context, id uuid.UUID) (Item, error) {
    m.mu.RLock()
    defer m.mu.RUnlock()

    item, ok := m.items[id]
    if !ok {
        return Item{}, ErrNotFound
    }
    return item, nil
}

func (m *Memory) Create(_ context.Context, name string, price float64) (Item, error) {
    m.mu.Lock()
    defer m.mu.Unlock()

    now := time.Now()
    item := Item{
        ID:        uuid.New(),
        Name:      name,
        Price:     price,
        CreatedAt: now,
        UpdatedAt: now,
    }
    m.items[item.ID] = item
    return item, nil
}

func (m *Memory) Update(_ context.Context, id uuid.UUID, name string, price float64) (Item, error) {
    m.mu.Lock()
    defer m.mu.Unlock()

    item, ok := m.items[id]
    if !ok {
        return Item{}, ErrNotFound
    }
    if name != "" {
        item.Name = name
    }
    if price > 0 {
        item.Price = price
    }
    item.UpdatedAt = time.Now()
    m.items[id] = item
    return item, true
}

func (m *Memory) Delete(_ context.Context, id uuid.UUID) error {
    m.mu.Lock()
    defer m.mu.Unlock()

    if _, ok := m.items[id]; !ok {
        return ErrNotFound
    }
    delete(m.items, id)
    return nil
}
```

Notice the `_` for unused `context.Context` parameters — the in-memory store doesn't need context
cancellation, but it must satisfy the interface.

### Sentinel errors

```go
// internal/store/errors.go
package store

import "errors"

var (
    ErrNotFound = errors.New("store: item not found")
)
```

Using `errors.Is` / `errors.As` compatible sentinel errors lets handlers distinguish "not found"
(404) from genuine database failures (500) without string-matching error messages.

### The Postgres repository

```go
// internal/store/postgres.go
package store

import (
    "context"
    "database/sql"
    "errors"
    "time"

    "github.com/google/uuid"
)

type PostgresRepo struct {
    db *sql.DB
}

func NewPostgresRepo(db *sql.DB) *PostgresRepo {
    return &PostgresRepo{db: db}
}

func (r *PostgresRepo) List(ctx context.Context) ([]Item, error) {
    rows, err := r.db.QueryContext(ctx,
        `SELECT id, name, price, created_at, updated_at
         FROM items
         WHERE deleted_at IS NULL
         ORDER BY created_at DESC`)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var items []Item
    for rows.Next() {
        var item Item
        if err := rows.Scan(&item.ID, &item.Name, &item.Price, &item.CreatedAt, &item.UpdatedAt); err != nil {
            return nil, err
        }
        items = append(items, item)
    }
    if err := rows.Err(); err != nil {
        return nil, err
    }
    return items, nil
}

func (r *PostgresRepo) Get(ctx context.Context, id uuid.UUID) (Item, error) {
    var item Item
    err := r.db.QueryRowContext(ctx,
        `SELECT id, name, price, created_at, updated_at
         FROM items
         WHERE id = $1 AND deleted_at IS NULL`, id).
        Scan(&item.ID, &item.Name, &item.Price, &item.CreatedAt, &item.UpdatedAt)
    if errors.Is(err, sql.ErrNoRows) {
        return Item{}, ErrNotFound
    }
    if err != nil {
        return Item{}, err
    }
    return item, nil
}

func (r *PostgresRepo) Create(ctx context.Context, name string, price float64) (Item, error) {
    var item Item
    err := r.db.QueryRowContext(ctx,
        `INSERT INTO items (name, price)
         VALUES ($1, $2)
         RETURNING id, name, price, created_at, updated_at`,
        name, price).
        Scan(&item.ID, &item.Name, &item.Price, &item.CreatedAt, &item.UpdatedAt)
    if err != nil {
        return Item{}, err
    }
    return item, nil
}

func (r *PostgresRepo) Update(ctx context.Context, id uuid.UUID, name string, price float64) (Item, error) {
    // Build a dynamic UPDATE — only set fields that were provided
    var item Item
    err := r.db.QueryRowContext(ctx,
        `UPDATE items
         SET name   = CASE WHEN $2 != '' THEN $2 ELSE name END,
             price  = CASE WHEN $3 > 0  THEN $3 ELSE price END,
             updated_at = now()
         WHERE id = $1 AND deleted_at IS NULL
         RETURNING id, name, price, created_at, updated_at`,
        id, name, price).
        Scan(&item.ID, &item.Name, &item.Price, &item.CreatedAt, &item.UpdatedAt)
    if errors.Is(err, sql.ErrNoRows) {
        return Item{}, ErrNotFound
    }
    if err != nil {
        return Item{}, err
    }
    return item, nil
}

func (r *PostgresRepo) Delete(ctx context.Context, id uuid.UUID) error {
    res, err := r.db.ExecContext(ctx,
        `UPDATE items SET deleted_at = now() WHERE id = $1 AND deleted_at IS NULL`, id)
    if err != nil {
        return err
    }
    n, _ := res.RowsAffected()
    if n == 0 {
        return ErrNotFound
    }
    return nil
}
```

A few patterns worth noting:

- **`RETURNING`** on `INSERT` and `UPDATE` avoids a second query. Postgres returns the full
  row after the mutation, including server-generated values like `id` and `created_at`.
- **`CASE WHEN` in the `UPDATE`** preserves existing values when the caller passes zero-values.
  If the caller sends `name = ""` and `price = 0`, neither field changes. If they send a positive
  price but no name, only the price updates. This matches the partial-update semantics of the
  original `Memory.Update` method.
- **Soft-delete via `UPDATE`** instead of `DELETE` ensures `deleted_at` records the deletion
  timestamp while keeping the row available for audit or recovery.
- **`sql.ErrNoRows` → `ErrNotFound`** maps the driver-level sentinel to our domain error,
  keeping handler logic clean.

### Inject the dependency

Now `cmd/api/main.go` accepts a `Repository` interface instead of `*store.Memory`:

```go
// cmd/api/main.go (excerpt)
package main

import (
    "context"
    "log"
    "net/http"
    "os"
    "os/signal"
    "time"

    "github.com/dhanifudin/go-api-demo/internal/handlers"
    "github.com/dhanifudin/go-api-demo/internal/middleware"
    "github.com/dhanifudin/go-api-demo/internal/store"

    "github.com/go-chi/chi/v5"
)

func main() {
    port := os.Getenv("PORT")
    if port == "" {
        port = "8080"
    }

    dsn := os.Getenv("DATABASE_URL")
    if dsn == "" {
        dsn = "postgres://app:secret@localhost:5432/app?sslmode=disable"
    }

    db, err := store.Open(dsn)
    if err != nil {
        log.Fatalf("database: %v", err)
    }
    defer db.Close()

    repo := store.NewPostgresRepo(db)
    h := handlers.NewItemHandler(repo)

    r := chi.NewRouter()
    r.Use(middleware.Recovery)
    r.Use(middleware.Logging)

    r.Get("/health", func(w http.ResponseWriter, r *http.Request) {
        writeJSON(w, http.StatusOK, map[string]string{"status": "ok"})
    })

    r.Route("/items", func(r chi.Router) {
        r.Get("/", h.List)
        r.Get("/{id}", h.Get)
        r.Post("/", h.Create)
        r.Put("/{id}", h.Update)
        r.Delete("/{id}", h.Delete)
    })

    srv := &http.Server{
        Addr:         ":" + port,
        Handler:      r,
        ReadTimeout:  5 * time.Second,
        WriteTimeout: 10 * time.Second,
        IdleTimeout:  15 * time.Second,
    }

    go func() {
        log.Printf("listening on :%s", port)
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatalf("server error: %v", err)
        }
    }()

    ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt)
    defer stop()

    <-ctx.Done()
    log.Println("shutting down gracefully...")

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Fatalf("shutdown error: %v", err)
    }
    log.Println("server stopped")
}
```

The handler signature changes from `*store.Memory` to `store.Repository` — but the handler
implementation itself barely changes:

```go
// internal/handlers/items.go (updated)
package handlers

import (
    "encoding/json"
    "errors"
    "net/http"

    "github.com/dhanifudin/go-api-demo/internal/store"

    "github.com/go-chi/chi/v5"
    "github.com/google/uuid"
)

type ItemHandler struct {
    store store.Repository  // was *store.Memory
}

func NewItemHandler(s store.Repository) *ItemHandler {
    return &ItemHandler{store: s}
}

func (h *ItemHandler) List(w http.ResponseWriter, r *http.Request) {
    items, err := h.store.List(r.Context())
    if err != nil {
        writeError(w, http.StatusInternalServerError, "failed to list items")
        return
    }
    writeJSON(w, http.StatusOK, items)
}

func (h *ItemHandler) Get(w http.ResponseWriter, r *http.Request) {
    id, err := uuid.Parse(chi.URLParam(r, "id"))
    if err != nil {
        writeError(w, http.StatusBadRequest, "invalid id format")
        return
    }
    item, err := h.store.Get(r.Context(), id)
    if errors.Is(err, store.ErrNotFound) {
        writeError(w, http.StatusNotFound, "item not found")
        return
    }
    if err != nil {
        writeError(w, http.StatusInternalServerError, "failed to get item")
        return
    }
    writeJSON(w, http.StatusOK, item)
}

// Create, Update, Delete follow the same pattern:
// - uuid.Parse the id from the URL
// - call h.store.Xxx(r.Context(), ...)
// - check store.ErrNotFound for 404
// - handle other errors as 500
```

The key differences from the in-memory version:

- `h.store.List(r.Context())` — passes the request context for cancellation
- `uuid.Parse()` — validates the ID before hitting the database
- `errors.Is(err, store.ErrNotFound)` — maps sentinel errors to HTTP status codes
- Error branches for database failures (500) — in-memory store never returned errors

You now have a fully pluggable data layer. Flip between `store.NewMemory()` and
`store.NewPostgresRepo(db)` depending on whether you're running unit tests or full-stack
integration tests.

## Writing repository tests against Postgres

Testing database code is the classic integration-test dilemma: you need a real Postgres to get
meaningful confidence, but spinning one up per test is slow and fragile.

Two approaches solve this:

| Approach | Setup time | Isolation | Best for |
|----------|-----------|-----------|----------|
| **Testcontainers** | ~5-10 s per suite | One container per suite | CI pipelines, realistic tests |
| **`txdb`** (transactional) | ~0 ms per test | Each test in its own tx | Fast local feedback |

We'll use **Testcontainers** — it starts a real Postgres container, runs your suite against it,
then tears everything down. The overhead is a few seconds per test suite, not per test, making
it practical for both local development and CI.

### Installing Testcontainers

```bash
go get github.com/testcontainers/testcontainers-go
go get github.com/testcontainers/testcontainers-go/modules/postgres
```

### Test setup helper

```go
// internal/store/postgres_test.go
package store

import (
    "context"
    "log"
    "os"
    "testing"
    "time"

    "github.com/testcontainers/testcontainers-go"
    "github.com/testcontainers/testcontainers-go/modules/postgres"
    "github.com/testcontainers/testcontainers-go/wait"
)

var testDB *sql.DB

func TestMain(m *testing.M) {
    ctx := context.Background()

    container, err := postgres.Run(ctx,
        "postgres:16-alpine",
        postgres.WithDatabase("testdb"),
        postgres.WithUsername("test"),
        postgres.WithPassword("test"),
        testcontainers.WithWaitStrategy(
            wait.ForLog("database system is ready to accept connections").
                WithOccurrence(2).
                WithStartupTimeout(30*time.Second),
        ),
    )
    if err != nil {
        log.Fatalf("start postgres: %v", err)
    }

    dsn, err := container.ConnectionString(ctx, "sslmode=disable")
    if err != nil {
        log.Fatalf("connection string: %v", err)
    }

    testDB, err = Open(dsn)
    if err != nil {
        log.Fatalf("open db: %v", err)
    }
    defer testDB.Close()

    // Run migrations
    if err := runMigrations(dsn); err != nil {
        log.Fatalf("migrate: %v", err)
    }

    code := m.Run()

    if err := container.Terminate(ctx); err != nil {
        log.Fatalf("terminate container: %v", err)
    }

    os.Exit(code)
}

func runMigrations(dsn string) error {
    // Use golang-migrate as a library:
    // migrate.New("file://../migrations", dsn)
    // Or, for simplicity, embed the SQL:
    migrations := []string{
        `CREATE EXTENSION IF NOT EXISTS pgcrypto`,
        `CREATE TABLE IF NOT EXISTS items (
            id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
            name VARCHAR(255) NOT NULL,
            price NUMERIC(10,2) NOT NULL CHECK (price > 0),
            created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
            updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
            deleted_at TIMESTAMPTZ
        )`,
    }
    for _, m := range migrations {
        if _, err := testDB.ExecContext(context.Background(), m); err != nil {
            return err
        }
    }
    return nil
}
```

`TestMain(m *testing.M)` is Go's test-suite-level hook. It runs once before any test function:
start Postgres, run migrations, then execute all tests in the package. After the last test, it
terminates the container. The result is one container lifecycle per package, not per test.

### Table-driven repository tests

```go
func TestRepo_CreateAndGet(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    item, err := repo.Create(ctx, "Mechanical Keyboard", 149.99)
    if err != nil {
        t.Fatalf("create: %v", err)
    }
    if item.ID == uuid.Nil {
        t.Error("expected non-zero UUID")
    }
    if item.Name != "Mechanical Keyboard" {
        t.Errorf("expected 'Mechanical Keyboard', got %s", item.Name)
    }
    if item.Price != 149.99 {
        t.Errorf("expected 149.99, got %f", item.Price)
    }

    got, err := repo.Get(ctx, item.ID)
    if err != nil {
        t.Fatalf("get: %v", err)
    }
    if got.ID != item.ID {
        t.Errorf("expected ID %s, got %s", item.ID, got.ID)
    }
}

func TestRepo_Get_NotFound(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    _, err := repo.Get(ctx, uuid.New())
    if !errors.Is(err, ErrNotFound) {
        t.Errorf("expected ErrNotFound, got %v", err)
    }
}

func TestRepo_List(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    // Seed two items
    _, _ = repo.Create(ctx, "Item A", 10.00)
    _, _ = repo.Create(ctx, "Item B", 20.00)

    items, err := repo.List(ctx)
    if err != nil {
        t.Fatalf("list: %v", err)
    }
    if len(items) < 2 {
        t.Errorf("expected at least 2 items, got %d", len(items))
    }
}

func TestRepo_Update_Partial(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    item, _ := repo.Create(ctx, "Original Name", 50.00)

    // Update only the price
    updated, err := repo.Update(ctx, item.ID, "", 75.00)
    if err != nil {
        t.Fatalf("update: %v", err)
    }
    if updated.Name != "Original Name" {
        t.Errorf("expected name unchanged, got %s", updated.Name)
    }
    if updated.Price != 75.00 {
        t.Errorf("expected price 75.00, got %f", updated.Price)
    }

    // Update only the name
    updated, err = repo.Update(ctx, item.ID, "New Name", 0)
    if err != nil {
        t.Fatalf("update: %v", err)
    }
    if updated.Name != "New Name" {
        t.Errorf("expected 'New Name', got %s", updated.Name)
    }
    if updated.Price != 75.00 {
        t.Errorf("expected price unchanged, got %f", updated.Price)
    }
}

func TestRepo_SoftDelete(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    item, _ := repo.Create(ctx, "Delete Me", 5.00)

    err := repo.Delete(ctx, item.ID)
    if err != nil {
        t.Fatalf("delete: %v", err)
    }

    // Confirm it's gone via Get
    _, err = repo.Get(ctx, item.ID)
    if !errors.Is(err, ErrNotFound) {
        t.Errorf("expected ErrNotFound after soft-delete, got %v", err)
    }

    // Double-delete should also return ErrNotFound
    err = repo.Delete(ctx, item.ID)
    if !errors.Is(err, ErrNotFound) {
        t.Errorf("expected ErrNotFound on double-delete, got %v", err)
    }
}

func TestRepo_Create_InvalidPrice(t *testing.T) {
    repo := NewPostgresRepo(testDB)
    ctx := context.Background()

    _, err := repo.Create(ctx, "Bad Item", -5.00)
    if err == nil {
        t.Error("expected error for negative price")
    }
}
```

These tests exercise the real Postgres, real constraints (`CHECK (price > 0)`), and real
behaviour (UUID generation, `RETURNING` clauses). They catch integration bugs that unit tests
against `Memory` would miss — like forgetting a `WHERE deleted_at IS NULL` clause or mishandling
the `NUMERIC(10,2)` → `float64` conversion.

## Wiring the repository into the API handlers

Here's the best part: if you built your handlers against the `Repository` interface from the
start, **you change nothing in your handler code**. The only change is in `main.go`:

```go
// Before (in-memory)
s := store.NewMemory()
h := handlers.NewItemHandler(s)

// After (Postgres)
db, _ := store.Open(os.Getenv("DATABASE_URL"))
defer db.Close()
repo := store.NewPostgresRepo(db)
h := handlers.NewItemHandler(repo) // ← same constructor, different impl
```

Run the full test suite — handler tests still pass against the `Memory` store (fast, no Docker),
and `PostgresRepo` tests validate the real database behaviour:

```bash
go test ./...
# ok   github.com/dhanifudin/go-api-demo/internal/handlers   0.012s
# ok   github.com/dhanifudin/go-api-demo/internal/middleware  0.008s
# ok   github.com/dhanifudin/go-api-demo/internal/store       5.234s
```

This is the power of the Repository pattern: fast unit tests for handler logic, integration
tests for database behaviour, and the same handler code for both.

## Running the full stack with Docker Compose

Now we tie it all together. Your `docker-compose.yml` from the [Docker primer](/blog/containers-and-docker)
already defined a `db` service with PostgreSQL. We'll enhance it to run migrations on startup
so the database is always in sync with the application code.

```yaml
services:
  api:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgres://app:secret@db:5432/app?sslmode=disable
    depends_on:
      db:
        condition: service_healthy
      migrate:
        condition: service_completed_successfully

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: app
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app"]
      interval: 2s
      timeout: 3s
      retries: 10

  migrate:
    image: migrate/migrate:v4.17.0
    volumes:
      - ./migrations:/migrations
    entrypoint:
      [
        "migrate",
        "-database", "postgres://app:secret@db:5432/app?sslmode=disable",
        "-path", "/migrations",
        "up"
      ]
    depends_on:
      db:
        condition: service_healthy

volumes:
  pgdata:
```

Key additions:

- **`migrate` service** — uses the official `migrate/migrate` Docker image. It runs `migrate up`
  against the database and exits. The `api` service won't start until it completes successfully
  (`condition: service_completed_successfully`).
- **Health check interval shortened** — `2s` instead of `5s` so the whole stack starts faster.
  In production you'd keep longer intervals to reduce load.
- **`pgdata` volume** persists data across `docker compose down`. Without it, your dev database
  resets on every restart.

Start the stack:

```bash
docker compose up --build
# db-1       | database system is ready to accept connections
# migrate-1  | 1/u create_items_table (2.341ms)
# api-1      | listening on :8080
```

Exercise the API against a real database:

```bash
curl -s -X POST http://localhost:8080/items \
  -H "Content-Type: application/json" \
  -d '{"name":"Mechanical Keyboard","price":149.99}' | jq
# {
#   "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
#   "name": "Mechanical Keyboard",
#   "price": 149.99,
#   "created_at": "2026-07-14T10:30:00Z",
#   "updated_at": "2026-07-14T10:30:00Z"
# }

# Restart the container
docker compose restart api

# The data is still there
curl -s http://localhost:8080/items | jq
# [{... same item, same UUID ...}]
```

No more wiping data on restart. The API connects to Postgres, Postgres persists data to a
named volume, and migrations run automatically on every `docker compose up`.

## Where to go next

You've taken the Go API from a single-instance prototype to a persistent, testable service backed
by PostgreSQL. The repository pattern you set up here will serve you well as you grow the
codebase — it keeps business logic decoupled from infrastructure, makes testing fast and
focused, and lets you swap backends without touching handlers.

### What we covered

- PostgreSQL schema design for the `items` domain (UUIDs, NUMERIC, TIMESTAMPTZ, soft deletes)
- Versioned migrations with `golang-migrate` (up/down scripts, Makefile helpers)
- Connecting to Postgres from Go with `pgx` and `database/sql`
- The Repository interface pattern with `context.Context` support
- Integration tests with Testcontainers (real Postgres, isolated per suite)
- Docker Compose with automatic migration on startup

### What to explore next

- **Containerise and deploy.** The [Containers and Docker](/blog/containers-and-docker) tutorial
  shows how to package this API into a multi-stage Docker image — an 8 MB `FROM scratch` container
  that starts in milliseconds. Combine it with the Docker Compose setup from this post for a
  complete local dev environment.

- **Orchestrate with Kubernetes.** The [Kubernetes for developers](/blog/kubernetes-for-developers)
  guide translates this Compose stack into Kubernetes Deployments, Services, ConfigMaps, and
  Secrets — ready for a multi-node cluster.

- **Automate with CI/CD.** The [CI/CD pipelines with GitHub Actions](/blog/ci-cd-pipelines-with-github-actions)
  post closes the loop: automate builds, Docker image pushes, and Kubernetes deployments on every
  push to main.

- **Extend the schema.** Add more domain tables (orders, users, categories), foreign keys, and
  more complex queries. Each new table gets its own migration file.

- **Add query filtering.** Build `GET /items?name=keyboard&min_price=10&max_price=200` with
  dynamic WHERE clauses in the repository.

- **Connection pooling in production.** For production workloads, tune `MaxOpenConns`,
  `MaxIdleConns`, and consider a standalone pooler like PgBouncer in front of your Postgres
  cluster.

- **Observability.** Add OpenTelemetry tracing to your repository methods to get span-level
  visibility into database query latency.

The stack you've built — Go + chi + pgx + Postgres + golang-migrate — is the foundation for
countless production services. From here, containerise it, deploy it, and keep building.
